# Plan: Bring Profile API Into Spec Compliance

## Goal

Implement the missing grading requirements and replace the current non-spec `GET /api/profiles?name=...` behavior with the required collection and record lookup endpoints.

## Chosen Direction

- Treat the grading spec as the source of truth.
- Replace the current `GET /api/profiles` name-lookup/create flow with a list endpoint that supports optional filters.
- Add `GET /api/profiles/{id}` alongside the existing `DELETE /api/profiles/{id}`.
- Keep `POST /api/profiles` as the only create/enrich entrypoint.
- Update docs/UI copy so the app no longer advertises the old lookup behavior.

## Implementation Steps

1. Refactor shared profile-route helpers in `app/api/profiles/route.js` and `app/api/profiles/[id]/route.js`.
   - Extract or standardize reusable pieces for CORS, database access, id normalization, document formatting, and error responses.
   - Keep the error envelope consistent everywhere: `{ "status": "error", "message": "..." }`.

2. Rework `GET /api/profiles` in `app/api/profiles/route.js`.
   - Remove the current dependency on the `name` query parameter.
   - Parse optional `gender`, `country_id`, and `age_group` query params case-insensitively.
   - Build a Mongo filter from only the supplied params.
   - Return `200` with the exact collection response shape:
     - `status`
     - `count`
     - `data` as an array
   - Return list items in the reduced shape required by the prompt (`id`, `name`, `gender`, `age`, `age_group`, `country_id`).

3. Keep `POST /api/profiles` compliant while preserving idempotency.
   - Continue normalizing the submitted name.
   - Continue reusing an existing profile when the normalized name already exists.
   - Continue creating UUID v7 `id` values and UTC ISO 8601 `created_at` timestamps.
   - Verify the stored document still contains all required aggregated fields.

4. Add `GET /api/profiles/{id}` in `app/api/profiles/[id]/route.js`.
   - Reuse id validation.
   - Query by public `id`.
   - Return `200` with the full profile payload when found.
   - Return `404` with the standard error envelope when missing.
   - Preserve `DELETE /api/profiles/{id}` behavior and update `OPTIONS` to advertise both `GET` and `DELETE`.

5. Tighten upstream validation and error messaging in `lib/enrichment.js`.
   - Keep the three concurrent API calls.
   - Validate the exact edge cases required by the prompt:
     - Genderize `gender: null`
     - Genderize `count: 0`
     - Agify `age: null`
     - Nationalize with no country data
   - Change each invalid upstream case to throw a service-specific message in the exact required form:
     - `Genderize returned an invalid response`
     - `Agify returned an invalid response`
     - `Nationalize returned an invalid response`
   - Ensure route handlers surface those as `502` without inserting anything.

6. Update automated tests to match the grading contract.
   - Replace the old `GET /api/profiles?name=...` tests in `test/profiles-route.test.mjs` with collection-list/filter tests.
   - Add coverage for:
     - no-filter list response
     - case-insensitive filtering
     - exact response shape with `count`
     - `POST` idempotency and create flow
     - `GET /api/profiles/{id}` success and `404`
     - `DELETE /api/profiles/{id}` success and `404`
     - `OPTIONS` headers for both route files
   - Expand `test/enrichment.test.mjs` for the required invalid upstream payload cases and exact error messages.

7. Update project-facing documentation and demo copy.
   - Revise `README.md` to document the four required endpoints and remove the obsolete name-lookup semantics.
   - Update `app/page.js` so the homepage examples and behavior bullets reflect the spec-compliant endpoints.

## Validation Plan

- Run the focused route and enrichment tests first to verify the contract changes.
- Run the full test suite after route/test updates.
- If the live DB test still depends on external connectivity, treat it separately from the API contract work and avoid changing it unless necessary for this task.

## Expected Outcome

- The API surface matches the required four endpoints exactly.
- The collection and record lookup endpoints return the expected shapes.
- Upstream invalid data returns the exact `502` messages required by the grader.
- Existing `POST`/`DELETE` behavior remains correct while the old non-spec `GET ?name=` path is removed.
